# 言語化

## 前提

文字には2つの用途がある。

- **情報伝達** — 正確かつ効率的に、情報を伝達・記録する。
- **自己表現** — その瞬間の自分の感情を、ただありのままに書く。

## 設計思想の違い

| ツール | 大切にしているもの |
|---|---|
| Notion | 情報の整理と共有 |
| Obsidian | 知識の蓄積と接続 |
| VSCode | 開発の効率と拡張性 |
| 縦書きエディター | 小説を書きやすくする |
| **あと、** | **書き手の感情表現とその肯定** |

あと、は信念系プロダクト。課題の解決ではなく、共感できる思想の提案で価値提供を行う。（例：万年筆）

## 価値の違い

- **Notion** : ビジネスツールとしての便利さ。メニューにはミーティングやプロジェクト管理が並び、必要なアクションにすぐ辿り着ける。
- **Obsidian** : 知識の繋がりとカスタム性。グラフやリンクを中心に、ユーザーが情報を自由に活用できる。
- **VSCode** : 開発体験。あくまで目的はコーディングであり、文字は「実行されるコード」として扱われる。
- **縦書きエディター** : 日本語での執筆のしやすさ。文字の書きやすさという機能性の軸にいる。
- **あと、** : 書き手が自身の感情表現を愛せること。機能より見た目のカスタム性に重きを置き、文章を一つの作品のように見せる。あらゆる情緒を受容し、書き手が自分の言葉を自分のものだと感じ、それを愛せるような仕掛けにする。

## ターゲットの違い

- 対象内
  - 日記、エッセイ、ポエム、手紙、思考の言語化をする人
  - 「書くこと」自体に心地よさや意味を感じている人
  - 既存ツール（Notion、Obsidian）の見た目やUIに「なんか違う」と萎えたことがある人
  - ビジネスツールの空気感の中でポエムを書きたくならない人
- 対象外
  - 情報伝達としての文字を重視する人
  - 情報整理やナレッジ管理が目的の人
  - 機能の多さやプラグインの豊富さを求める人

## 体験の仮説

**「書く → 自分の言葉が自分好みに表示される → 自分の言葉だと感じられる → 自分の表現に愛着が湧く → 書きたくなる」**

## 機能要件の仮説

- **UIカスタム**: フォント・行間・サイズ・背景を自分好みに細かく調整できる → 自分の言葉であると感じられる
- **日本語組版**: 書いたものが小説作品のように見える → 自分の言葉に愛着が湧く
- **認知コスト** : UIはできる限り削り、文章の表示を最優先する → 自分の情動が邪魔されない
- **いつもそこに**: どんなデバイスでも、どんな通信状況でも、同じ作業状況のままある　→　自分の認知と違わないことによる安心感

## LPで伝えるべきこと

信念系のプロダクトに必要なのは機能の説明ではなくマニフェストと体験。

- **言葉で説明するのではなく、実際に使わせる** — 登録不要で書ける体験版をLPに組み込み、このループを1回まわしてもらう
- 体験の前後に、信念（なぜ情緒を大切にするのか）を添える

## キャッチコピー候補

### 世界観・肯定系

- それは私を忘れぬように
- 主人公、わたし。
- 人生は、作品だ。
- 拝啓、未来の私。
- 文字は心。
- わたしを書く、だけの部屋。
- わたしを書く、それだけ。
- （確定）僕を書くためのエディター。

### 情緒・行為系

- 心を映す
- 気持ちを文字に
- 感情的に書く
- 書きたい。
- 体験を書く。

### 挑発・衝動系

- 考えるな、書け。
- 文字は情報じゃない。

### 組み合わせ案

- 主人公、わたし。— わたしを書くための書斎
- 考えるな、書け。— 心を映す執筆ツール
- 今日の気持ちを、明日の自分が好きになる。
- 気持ちを書く自分を、好きになる。
- 泣いた日も、嬉しい日も、ぜんぶ美しく残る。

### 方向性メモ

- 「気持ちを書く自分のことが好きになる」が言語化の核
- 機能説明（マークダウン、AI等）はキャッチコピーに不要
- 信念系なので、説明ではなく共鳴を狙う
- ひらがな表記が世界観に合う（漢字はビジネス感が出る）
- 「部屋」は空間・居場所・執筆ツールのニュアンスを兼ねる
