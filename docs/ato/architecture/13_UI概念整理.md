# UI概念整理

## 背景

LP用のテスト画面として4ペルソナのエディタ画面(20代男性/女性、50代女性、70代男性)をFigma Makeで生成した結果、同じプロダクトとは思えないほど画面が違い、実装可能なシステムに落とし込めないことが判明した。

「あと、」は書き手が自分だけの空間を作れることを価値にするプロダクトである以上、カスタム性の幅広さは思想と表裏一体だが、無限のカスタム余地を抱えるとUX・実装の両方が破綻する。

どこを統一し、どこをカスタムにするかの線引きと整理自体が「あと、」が提供するべき価値になる。

## 思想

- ユーザーの世界を破壊してはならない
- 見た目や雰囲気はユーザーに委ねる
- エディターを成立させる最低限の要素のみ固定する
- 各要素を**不変/可変**で分類する
- ユーザーの認知に基づき設定項目を分類する(ユーザーが設定で迷わないよう実装ではなく使用目的で分ける)

## 分類

### 不変の項目

- 全画面がエディタ(UIで執筆を邪魔しない)
- メニューボタンが上下左右のどこかに必ず存在する(メニューボタンを全アクションの起点とし、アクション不能状態を防ぐ)
- テキスト選択色・IME変換色などはデバイスに従う(全ユーザーが認知不可抑制のため)
- 画面内に「あと、」ロゴや記号は出さない(ユーザーの世界観を壊すため)

### 可変の項目

- **文字**
  - フォント
  - サイズ
  - アキ(行間・文字列間・エディター幅など)
  - 文字色
  - 書字方向
- **紙**
  - 背景(単色・パターン・画像など)
  - 罫線有無
- **音**
  - BGM
  - SE
- **機能**
  - ※メニュー/AI/ファイルなど
  - 機能オンオフ
  - ボタン位置
  - ボタンデザイン
  - 展開デザイン
  - 展開方向
  - 機能の挙動パラメータ(機能がアクティブな場合のみ設定可能)
- **プリセット**
  - ※水準が違うが利便性のため追加
  - テーマ(カスタム・ダーク・ライト)

### 将来的な追加項目への対応方針

- **感覚系は五感ごとに並列項目を立てる**
  - 現状は視覚(文字・紙)と聴覚(音)のみ。触覚(ハプティクス等)が必要になれば音と並列で追加する。
- **非機能要件(同期・バックアップ・自動保存・通信等)は原則不変とする**
  - 本プロダクトは「見た目のカスタム」を趣旨とするため、非機能要件は設定不可。
- **機能項目は各機能の詳細パラメータまで含む**
  - 「その機能がアクティブな場合のみ詳細まで設定可能」という制約で制御する。
- **複合モード(例: 集中モード)は既存項目で表現する**
  - 通知ミュート=機能、遮音=音、のように "操作対象" で既存項目に落とす。

## 命名

### UI

- `Editor`
  - 全画面の入力エリア、現在は入力不可でコンテンツのみを配置している
- `{機能名}Button`
  - メニューやファイルなど各機能を幼児するトリガー、エディター上にユーザーが自由に配置。
- `{機能名}Panel`
  - Buttonによって開かれるコンテンツ。メニュー設定項目やファイル一覧を表示。

### CSS

`{UI名}`がないものは共通仕様とする。

- `{UI名}BgColor`
  - UIの背景色
- `{UI名}Border.Size/Border.Color/Border.Shadow`
  - UIの枠線の太さ・色・影
  - それぞれ左右上下を別々にUIで設定可能にする
